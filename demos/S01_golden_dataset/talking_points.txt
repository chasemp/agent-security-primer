KEY TAKEAWAY
A golden dataset is the test suite for a non-deterministic system.
Curate (input, expected_label) pairs once and any prompt change or
model swap can be measured against them. The dataset gives you two
signals: prompt regressions become numbers (75% -> 12% is a hard
delta), and label drift becomes visible (when the model consistently
disagrees with a label, audit the label, not the model).

BACKGROUND
Unit tests work because the function is deterministic: same input, same
output, every time. LLMs break that assumption. You can't write a unit
test that says "this prompt returns 'homesick' for this input" because
the model might say "Homesick.", "homesick", "the mood is homesick", or
flip to "Reflective" entirely if the prompt loses its label vocabulary.

The golden dataset is how you regain testability. You curate 8-50 pairs
of (input, expected_label) once, treat them as the contract, and score
any future prompt or model against them. The scoring is fuzzy
(case-insensitive whole-word match here), but the dataset is fixed.
When your accuracy drops from 75% to 12% after a prompt edit, that's a
regression you can act on — same kind of signal as a failing unit test
in a deterministic system, just expressed as a percentage.

This demo uses 8 letters from the camp_letters corpus (one per mood
category), deliberately mixing prototypical cases with multi-signal
ones. Two prompt versions: v1 with rich category descriptions, v2
stripped of the label vocabulary entirely.

WHY JSONL?
One JSON object per line. Each line is an independent test case, so
edits don't ripple. wc -l counts test cases, head -3 shows the first
three, jq slices the rest — no custom parser needed. Stream-friendly
(eval.py reads line-by-line). And it's the industry standard for ML
evals (OpenAI fine-tuning, HuggingFace datasets, pydantic-evals all
consume JSONL).

The format is simple but the contents are easy to make opaque. Treat
the file as plain text and inspect it before trusting it.

THE DEMO
The audience sees two payoffs from the same eval run:

PAYOFF 1 — Prompt regression (v1 vs v2).
  v1 (descriptions, full vocabulary): 6/8 correct (75%)
  v2 (no label vocabulary):           1/8 correct (12.5%)
  Same dataset, different score. The delta IS the regression signal.

PAYOFF 2 — Label drift (the v1 failures).
  Both prompts persistently miss the same two letters:
    Letter 3 — labeled "adventure" but reads as "food" (kid talks
               only about campfire cooking)
    Letter 8 — labeled "last_days" but reads as "friendship" (kid
               talks only about journal-trading and a cabin spider)
  Two prompts agreeing against the dataset means the dataset is the
  thing under suspicion, not the prompt. This is a feature, not a bug:
  golden datasets surface label drift the same way they surface prompt
  regressions. Audit the label, then either fix it or accept the
  ambiguity as a known edge case.

PRESENTER FLOW

Step 1 — Demystify the dataset (no API calls).

  # Show the file is just text, one entry per line
  wc -l demos/S01_golden_dataset/golden.jsonl

  # Pretty-print the first entry
  head -1 demos/S01_golden_dataset/golden.jsonl | jq

  # Show all the expected labels (sanity check we cover all 8)
  jq -r .expected demos/S01_golden_dataset/golden.jsonl | sort -u

  # Show just the inputs as plain text (drop the JSON wrapper)
  jq -r .input demos/S01_golden_dataset/golden.jsonl | head -20

  # Filter to one category
  jq 'select(.expected == "homesick")' \
    demos/S01_golden_dataset/golden.jsonl

Step 2 — Show the two prompt versions.

  cat demos/S01_golden_dataset/system_prompt_v1.txt
  cat demos/S01_golden_dataset/system_prompt_v2.txt

  # v1 names all 8 categories with descriptions (~700 chars).
  # v2 is one sentence: "Classify the dominant mood. One word."
  # Same task, drastically different specificity.

Step 3 — Run the eval against v1 (the good prompt).

  python scripts/eval.py demos/S01_golden_dataset/golden.jsonl \
    --prompt demos/S01_golden_dataset/system_prompt_v1.txt

  # Expect: 6/8 correct (75%). Look at the per-category breakdown.
  # The two failures are letters where the dataset label and the
  # surface lexical content disagree (campfire cooking, journaling).

Step 4 — Run the eval against v2 (the degraded prompt).

  python scripts/eval.py demos/S01_golden_dataset/golden.jsonl \
    --prompt demos/S01_golden_dataset/system_prompt_v2.txt

  # Expect: 1/8 correct (12.5%). Note WHAT v2 returns: Excited,
  # Joyful, Lighthearted, Reflective, Nostalgic. These are perfectly
  # reasonable mood words — they just don't belong to our rubric's
  # vocabulary. The model can't pick from a list it was never given.

Point out: v2 isn't broken in some abstract way. It's missing the
label list. Without it, the model defaults to its own vocabulary, and
our scorer (which expects "homesick", "adventure", etc.) misses every
output that uses synonyms. Without a golden dataset, this regression
is invisible — both prompts produce "reasonable looking" output. The
dataset is what makes "reasonable" measurable.

WHY THIS MATTERS
You can't prevent regressions you can't measure. Most LLM systems
have no eval harness — when someone tweaks a prompt, the only
verification is "I ran it on three examples and it looked fine."
That's wishful thinking.

A golden dataset is the smallest viable eval harness: 8-50 entries,
a couple hours to curate, then every prompt change and model swap is
testable. It's also the substrate for S02 (process vs outcome eval)
and S03 (consistency voting) — they add richer scoring on top.

Q&A PREPARATION

Q: 8 entries seems small. Won't accuracy be too noisy?
A: For a demo, 8 makes the regression visible. For production, you
   want 50-200. Diminishing returns past that. The point is to have
   *enough* signal to detect prompt-level changes.

Q: Why does v1 only get 6/8 on a curated dataset?
A: That's payoff 2. Both prompts agree the "adventure" letter is
   really about food and the "last_days" letter is really about
   friendship. When two independent prompts agree against the
   dataset, the label is suspect, not the prompt. In production
   golden-dataset work, this is how you discover that 5% of your
   labels are wrong without rereading every entry by hand.

Q: Why temperature=0?
A: Determinism for the eval. We want the dataset to measure prompt
   quality, not sampling noise. S03 (consistency voting) deliberately
   uses high temperature to measure noise — different question.

Q: Can the model see the expected label?
A: No. The model only sees the system prompt + the input letter.
   The expected label exists only in the JSONL and the scoring code.
