DEMO 17: TOKENOMICS — What Every Turn Actually Costs

KEY TAKEAWAY
Tokens are money. Two opt-in controls — prompt caching and thinking
token management — determine whether your agent loop burns 10x what
it needs to. Understanding the cost mechanics turns a "credit card
attached to a while loop" into a predictable line item.

This demo covers two cost levers:
  1. Prompt caching — the system prompt is re-read every turn.
     Without caching, you pay full price each time. With cache_control,
     turns 2+ read from cache at 90% discount.
  2. Thinking token accumulation — ThinkingBlocks stay in the messages
     array. Each turn's thinking tokens become input tokens for every
     subsequent turn. Costs compound quadratically.


PART 1: PROMPT CACHING

BACKGROUND
Every turn in the agent loop sends the full system prompt to the API.
The model re-reads it from scratch. At Opus rates ($15/MTok input),
a 2,000-token system prompt across 10 turns = 20,000 input tokens
just for the system prompt. That's $0.30 in system-prompt cost alone.

Prompt caching is OPT-IN. You must add a cache_control breakpoint
to the system prompt in the API call. Without it, every turn pays
full price. With it, the server caches the prompt on turn 1 and
subsequent turns read from cache.

THE PRICING
  - Cache write (turn 1):   1.25x the base input price
  - Cache read (turns 2+):  0.10x the base input price
  - No caching (default):   1.00x the base input price

THE MATH (10-turn agent loop, 2K system prompt)
  Without cache:  10 turns × 2,000 tokens × $15/MTok = $0.30
  With cache:     1 write × 2,000 × $18.75/MTok      = $0.0375
                  9 reads × 2,000 × $1.50/MTok        = $0.027
                  Total                                = $0.0645

  Savings: 78% on system prompt cost alone.

  At scale: 100 agents × 30 turns/day × $15/MTok = real money.

THE MINIMUM THRESHOLD — IT VARIES BY MODEL

Prompt caching requires a minimum prompt size. The threshold
varies by model tier, and this is worth understanding:

  ┌────────────────┬───────────┬──────────┬──────────────────┐
  │ Model          │ Min Tokens│ Input $/M│ Cache ROI        │
  ├────────────────┼───────────┼──────────┼──────────────────┤
  │ Sonnet 4.6     │ 2,048     │ $3.00    │ Moderate         │
  │ Haiku 4.5      │ 4,096     │ $1.00    │ Low (cheap model)│
  │ Opus 4.6       │ 4,096     │ $15.00   │ Massive          │
  └────────────────┴───────────┴──────────┴──────────────────┘

The insight: Haiku is so fast and cheap that the overhead of
maintaining a cache isn't worth it until the prompt is large.
But Opus has the SAME high threshold — and that's where caching
matters most because the per-token cost is 15x Haiku.

If your system prompt is below the threshold, cache_control is
accepted but caching silently does nothing. No error. No warning.
You're paying full price and don't know it.

HOW CACHING WORKS — PREFIX MATCHING

THE MENTAL MODEL
Think of the API request as a long string of text. The server
hashes it from the beginning, section by section. A cache_control
marker says "everything from byte 0 to HERE is a cacheable
section." On the next request, the server hashes the incoming
text from the start and compares: "do I already have this
prefix?" If yes, it reads from cache instead of reprocessing.

It's like a book. If you've already read chapters 1-5, and
someone hands you the same book with a new chapter 6, you
don't re-read chapters 1-5. You skip to chapter 6. The
cache_control marker is the bookmark that says "I've read
up to here."

THE SIMPLE CASE — JUST THE SYSTEM PROMPT

The simplest version: cache only the system prompt.

  Turn 1: [system_prompt CC] + user_msg
           ┌───────────────┐  ┌────────┐
           │ CACHE WRITE   │  │ process│
           │ (1.25x)       │  │ (1x)   │
           └───────────────┘  └────────┘

  Turn 2: [system_prompt CC] + user_msg + asst_T1 + tool_T1
           ┌───────────────┐  ┌─────────────────────────────┐
           │ CACHE READ    │  │ process fresh               │
           │ (0.1x)        │  │ (1x)                        │
           └───────────────┘  └─────────────────────────────┘

The system prompt is byte-for-byte identical on both turns.
The server sees the match and reads it from cache. Everything
after the cached prefix is processed fresh.

THE MOVING BREAKPOINT — CONVERSATION CACHING

Now the interesting part. What if we put cache_control on
the LATEST user message instead of just the system prompt?
The cached prefix grows each turn:

  Turn 1: [system CC] + [tools CC] + [user_1 CC]
           ┌──────────────────────────────────────┐
           │ CACHE WRITE — entire request cached   │
           └──────────────────────────────────────┘

  Turn 2: [system] + [tools] + [user_1] + [asst_1] + [user_2 CC]
           ┌────────────────────────────┐  ┌────────────────────┐
           │ CACHE READ — prefix match  │  │ NEW — process + CC │
           │ system+tools+user_1 same   │  │ extends the cache  │
           └────────────────────────────┘  └────────────────────┘

  Turn 3: [system] + [tools] + [user_1] + [asst_1] + [user_2] + [asst_2] + [user_3 CC]
           ┌──────────────────────────────────────────────────────┐  ┌──────────────────┐
           │ CACHE READ — everything up to user_2 matches cache   │  │ NEW — asst_2 +   │
           │                                                      │  │ user_3 + CC      │
           └──────────────────────────────────────────────────────┘  └──────────────────┘

On each turn:
  1. The server hashes from byte 0 and matches against cache
  2. Everything before the NEW content matches — read from cache
  3. Only the NEW content (latest assistant response + latest
     user message) is processed fresh
  4. The CC marker on the latest message EXTENDS the cache to
     include everything up to here

The cache grows. By turn 10, turns 1-9 are all cached. The
server only processes the latest exchange.

WHY THIS WORKS: IDENTICAL PREFIXES
The key insight: message content never changes once sent. Turn 1's
user message is the same on turn 1 as it is on turn 5. The
assistant's response from turn 1 is the same on turn 5. So the
prefix — everything before the latest exchange — is always an
exact match.

We move the CC marker forward each turn (strip it from prior
messages, add it to the latest). But the CONTENT of those prior
messages is unchanged. The server sees the same prefix and reads
it from cache. Only the latest exchange is new.

THE HASH-SECTION ANALOGY
Think of each CC marker as defining a hash section. The server
hashes each section independently:

  ┌─────────────┬──────────┬───────────────────────┬────────────┐
  │ system hash  │ tools    │ conversation hash     │ new content│
  │ (CC marker)  │ (CC)     │ (CC on latest msg)    │            │
  └─────────────┴──────────┴───────────────────────┴────────────┘
        ↓              ↓              ↓
    cache hit?     cache hit?     cache hit?
        ↓              ↓              ↓
      READ           READ           READ or WRITE

If the hash matches → read from cache (0.1x).
If the hash doesn't match → process and write to cache (1.25x).
New content after the last CC marker → always processed (1x).

This is why the 4-block limit matters: each CC marker creates a
hash section. More sections = more flexibility but the API caps
it at 4.

The cache is keyed on exact content — if you change one character
in the system prompt, its hash changes, and it's a cache miss.
This is also why you strip CC from old messages: you're not
changing their content, just removing the marker. The hash of
the CONTENT is unchanged, so the prefix still matches.

CACHING TOOL DEFINITIONS
Tool definitions (the tools array) are sent to the API every
turn — just like the system prompt. For an agent with 6 tools
and detailed schemas, that's easily 800-1500 tokens per turn,
reprocessed from scratch.

The fix: put cache_control on the LAST tool in the tools array.
The API caches system prompt + all tool definitions as a single
prefix block.

THE MATH (this demo's tool block)
  Our 6 tools serialize to ~3000 characters ≈ ~800 tokens.
  Over 10 turns on Opus ($15/MTok):

  Without tool caching:  10 × 800 × $15/MTok = $0.12
  With tool caching:     1 write × 800 × $18.75/MTok = $0.015
                         9 reads × 800 × $1.50/MTok  = $0.011
                         Total                        = $0.026

  Savings: 78% on tool definition cost.

  Production agents with 15-20 tools and complex schemas can
  have 3000-5000 tokens in tool definitions. At that scale,
  the savings are proportionally larger.

THE CODE CHANGE
  # Add cache_control to the last tool definition:
  tool_defs = [dict(d) for d in tool_definitions]
  tool_defs[-1]["cache_control"] = {"type": "ephemeral"}

CACHING CONVERSATION HISTORY
This is what Claude Code does — and it's the biggest win for
long-running agent loops. Put cache_control on the last content
block of every user and assistant message. This means each turn's
entire conversation prefix (system prompt + all prior messages)
is cached.

Without conversation caching:
  Turn 10 processes: system_prompt + tools + messages 1-9 + new message
                     ALL at full price

With conversation caching:
  Turn 10 reads: system_prompt + tools + messages 1-9 from cache (0.1x)
  Turn 10 processes: only the new message (1x)

For a 30-turn agent loop, turn 30 would reprocess ~29 turns of
history at full price without this. With conversation caching,
only the newest message is processed fresh.

Our agent.py --cache flag does all three layers:
  1. System prompt — cache_control on the system text block
  2. Tool definitions — cache_control on the last tool schema
  3. Conversation history — cache_control on the last tool_result

THE THREE LAYERS COMBINED (10-turn loop on Opus, this demo)
  System prompt:  ~2000 tokens
  Tool schemas:   ~800 tokens
  Static total:   ~2800 tokens per turn

  Without caching: 10 × 2800 × $15/MTok         = $0.42
  With caching:    1 write × 2800 × $18.75/MTok  = $0.053
                   9 reads × 2800 × $1.50/MTok   = $0.038
                   Total                          = $0.091

  Savings: $0.33 per run. That's 78% reduction on the static
  prefix alone — before conversation caching even kicks in.

  Add conversation caching and the savings grow with each turn,
  because the cached prefix includes all prior messages too.

THE 4-BLOCK LIMIT
The API allows at most 4 cache_control blocks per request. If
you exceed this, you get an error. This means you have to be
strategic about where you place your breakpoints:

  Block 1: System prompt
  Block 2: Last tool definition
  Block 3: Latest user message (tool results)
  Block 4: (spare — for future use or additional caching)

The gotcha: if you naively add cache_control to EVERY user
message in the conversation history, you'll exceed 4 blocks by
turn 3. The fix: only the MOST RECENT user message gets
cache_control. Strip it from prior messages before each API call.

Our agent.py --cache flag handles this automatically. On each
turn it removes cache_control from all prior user messages and
adds it to the latest one. The cached prefix grows: the API
caches everything up to the latest breakpoint, which includes
all prior messages.

This is a real constraint you'll hit in production. Document it,
test for it, and manage the breakpoints intentionally.

THE CACHE TTL — TWO OPTIONS

  ┌──────────┬──────────────┬───────────────────────────────────┐
  │ TTL      │ Write Cost   │ Best For                          │
  ├──────────┼──────────────┼───────────────────────────────────┤
  │ 5 min    │ 1.25x input  │ Active agent loops, rapid-fire    │
  │ (default)│              │ API calls. Cache refreshes free    │
  │          │              │ on each hit within the window.     │
  ├──────────┼──────────────┼───────────────────────────────────┤
  │ 1 hour   │ 2.00x input  │ Slower interactions: chat where   │
  │          │              │ users take >5 min to respond,      │
  │          │              │ agentic sub-tasks that run long,   │
  │          │              │ batch jobs that run more often     │
  │          │              │ than hourly.                       │
  └──────────┴──────────────┴───────────────────────────────────┘

The 5-minute default means: an active agent loop that calls the
API every few seconds stays cached indefinitely — each call
refreshes the TTL for free. A batch job that runs every 10
minutes loses the cache between runs and pays the write cost
each time. For that pattern, the 1-hour TTL at 2x write cost
saves money over repeated 5-minute cache misses.

To use 1-hour TTL:
  {"cache_control": {"type": "ephemeral", "ttl": "1h"}}

One more effect: cache hits are NOT deducted from your API rate
limits. Heavy caching can effectively increase your throughput
ceiling.


AUTOMATIC CACHING — THE PRACTICAL VERSION

Everything above shows EXPLICIT caching — you place cache_control
markers on individual content blocks, manage the 4-block limit,
strip markers from old messages. That's the transparent version:
you see exactly what gets cached and why.

The API also supports AUTOMATIC caching. One top-level field on
the request body and the system handles everything:

  {
    "model": "claude-haiku-4-5",
    "max_tokens": 4096,
    "cache_control": {"type": "ephemeral"},   ← this is it
    "system": system_prompt,
    "tools": tool_defs,
    "messages": messages,
  }

The system automatically places the cache breakpoint on the last
cacheable block and moves it forward as the conversation grows.
No manual marker management. No stripping old messages.

  Request 1: [system + user_1] ◀ auto cache point
             Everything written to cache.

  Request 2: [system + user_1 + asst_1 + user_2] ◀ auto cache point
             system+user_1 read from cache.
             asst_1+user_2 written to cache.

  Request 3: [system + ... + user_3] ◀ auto cache point
             Everything through user_2 read from cache.
             Only the newest exchange written.

COMBINING EXPLICIT AND AUTOMATIC

They work together. The strategy:
  - Explicit breakpoints for STABLE content (system prompt, tools)
  - Automatic caching for GROWING content (conversation history)

  {
    "cache_control": {"type": "ephemeral"},          ← auto for conversation
    "system": [{
        "type": "text",
        "text": system_prompt,
        "cache_control": {"type": "ephemeral"}       ← explicit for system
    }],
    "tools": tool_defs_with_cache_on_last,           ← explicit for tools
    "messages": messages,                             ← auto handles these
  }

Both share the 4-breakpoint limit. Explicit markers use slots
first; automatic uses whatever's left.

Edge cases:
  - If the auto target already has an explicit breakpoint with the
    same TTL, automatic is a no-op (harmless).
  - If TTLs conflict (explicit 1h, auto 5m on same block), you get
    a 400 error.
  - If all 4 slots are used by explicit markers, no room for
    automatic — you get a 400 error.

WHY SHOW BOTH IN THE DEMO
We show explicit caching first because it teaches the MECHANISM.
You see where the breakpoints go, why prefix matching works, and
what the 4-block limit means. Then we show automatic caching as
the practical follow-up: "now that you understand it, here's the
one-liner that handles it for you."

Automatic caching is simpler. But if it breaks — cache misses,
unexpected costs — you need to understand the explicit mechanics
to debug it. The transparent version isn't overhead; it's the
knowledge that makes the automatic version trustworthy.

PART 2: THINKING TOKEN ACCUMULATION

BACKGROUND
When extended thinking is enabled, the API returns ThinkingBlocks
in response.content. In a multi-turn agent loop, you append the
assistant's full response (including ThinkingBlocks) to the messages
array. On the next turn, ALL prior ThinkingBlocks are sent back
to the API as input tokens.

THE MECHANISM
  Turn 1:  input = system_prompt + user_msg                = 500 tokens
  Turn 2:  input = above + T1_thinking + T1_text + T1_tool = 700 tokens
  Turn 3:  input = above + T2_thinking + T2_text + T2_tool = 900 tokens
  ...
  Turn 30: input = all accumulated thinking + text + tools  = ???

Each turn's thinking tokens stay in the conversation forever.

THE COST IMPACT
  Simple tasks (~190 thinking tokens/turn):
    30 turns adds ~5,700 extra input tokens → modest cost

  Complex investigations (~2,000 thinking tokens/turn):
    30 turns adds ~60,000 extra input tokens → 30% of context window
    At Opus rates: 60K × $15/MTok = $0.90 in accumulated thinking

  Combined with a 2K system prompt (uncached):
    30 × 2,000 = 60K system prompt tokens = $0.90
    + 60K accumulated thinking             = $0.90
    Total overhead: $1.80 per investigation

THE DEMO FLOW
  # "We already saw thinking in Demo 8. Now let's see what it costs."
  # "Run with thinking, no cache — watch input tokens grow per turn"
  cat demos/17_tokenomics/task.txt | python scripts/agent.py \
      demos/17_tokenomics/system_prompt.txt \
      --tools demos/17_tokenomics/tools.py --thinking

  # "Same task with caching — system prompt is cheaper now"
  cat demos/17_tokenomics/task.txt | python scripts/agent.py \
      demos/17_tokenomics/system_prompt.txt \
      --tools demos/17_tokenomics/tools.py --thinking --cache

  Compare: the thinking tokens still accumulate (that's unavoidable
  without stripping), but the system prompt cost drops dramatically.
  The audience sees both levers in the same output.

THE MITIGATION
Strip ThinkingBlocks from response.content before appending to the
messages array:

  # Before appending to messages:
  content_without_thinking = [
      block for block in response.content
      if block.type != "thinking"
  ]

The model loses its own reasoning history but retains tool results
and text output. This is usually sufficient — the model reasons
fresh each turn based on the accumulated facts, not its prior
reasoning process.

Trade-off: verify stripping doesn't degrade output quality for your
use case before making it default. Some complex multi-step tasks
benefit from the model seeing its own prior reasoning.


PART 3: CIRCUIT BREAKER — TOKEN BUDGET

BACKGROUND
Caching reduces cost per turn. But the agent can still run away.
A confused model, a poison injection, or an ambiguous task can
cause the loop to burn through turns until max_turns stops it.
By then the damage is done.

A token budget circuit breaker is the financial hard stop. You
set a ceiling — say 5000 tokens — and the agent exits immediately
when cumulative tokens exceed it, regardless of whether the model
is done or wants more turns.

This is different from max_turns (which caps iterations). A token
budget caps SPEND directly. A 3-turn conversation with a huge
context can cost more than a 10-turn conversation with a small
context. max_turns doesn't catch that; a token budget does.

THE CODE
  result = run_agent(
      system_prompt, task, tools,
      max_tokens_budget=5000,   # hard stop at 5000 total tokens
  )

  if result.get("budget_exceeded"):
      print("Circuit breaker tripped — investigate why")

THE DEMO FLOW
  # "Set a tight budget and watch it trip"
  cat demos/17_tokenomics/task.txt | python scripts/agent.py \
      demos/17_tokenomics/system_prompt.txt \
      --tools demos/17_tokenomics/tools.py --budget 1500

  # The agent will trip the breaker after 1-2 turns because
  # the system prompt + tools + task already consume ~3000 tokens.
  # The audience sees: [BUDGET EXCEEDED — circuit breaker tripped]

  # "Now a reasonable budget"
  cat demos/17_tokenomics/task.txt | python scripts/agent.py \
      demos/17_tokenomics/system_prompt.txt \
      --tools demos/17_tokenomics/tools.py --budget 20000

  # Completes normally. The budget was headroom, not a constraint.

PART 4: TOKEN MINIMIZATION — Compress Before You Send

BACKGROUND
Every token that enters the context window costs money. Tool
results, API responses, log output — all of it gets tokenized
and billed. But most of that text has whitespace, formatting,
and structure that the model doesn't need to understand the
content.

Minimization is simple: compress the text before it becomes
tokens. Strip whitespace, compact JSON, remove blank lines.
The data is the same. The token count drops.

THE SIMPLE VERSION (20 lines of Python)

  # "Here's a verbose tool result — pretty-printed JSON."
  cat demos/17_tokenomics/verbose_result.txt | wc -c
  # → ~850 characters

  # "Run it through a dead-simple minifier."
  cat demos/17_tokenomics/verbose_result.txt | python scripts/minify.py
  # → ~480 characters, same data

  # "That's a 44% reduction. Same information, fewer tokens."

The minify.py script is 20 lines. It tries to parse as JSON
and re-serialize compactly. If it's not JSON, it strips
trailing whitespace and collapses blank lines. That's it.

  cat demos/17_tokenomics/verbose_result.txt | python scripts/minify.py
  # Before: 850 chars (~212 tokens)
  # After:  480 chars (~120 tokens)
  # Saved:  92 tokens (43%)

THE MATH
For a 10-turn agent loop on Opus where each tool result is
~200 tokens of verbose JSON:

  Without minification: 10 turns × 200 tokens × $15/MTok
  With minification:    10 turns × 114 tokens × $15/MTok
  Savings per run: ~$0.013

Small per run. But agents at scale — 100 agents × 30 turns ×
10 tool calls each = 30,000 tool results. At 86 tokens saved
per result, that's 2.58M tokens saved = $38.70 on Opus.

THE PRODUCTION VERSION: RTK

RTK (Rust Token Killer) is a production-grade tool that does
this at scale. It sits as a proxy between CLI commands and the
context window, automatically compressing output.

Real-world stats from RTK:
  - 11,168 commands processed
  - 20.2M tokens saved
  - 79% average efficiency
  - Top savings: find (-98%), git push (-91%), ls (-72%)

RTK does more than whitespace stripping:
  - Removes ANSI color codes
  - Filters redundant output (e.g., git status noise)
  - Truncates oversized results
  - Tracks savings analytics (rtk gain)

The principle is the same as our 20-line Python script. The
implementation is just faster and more comprehensive.

  # Show RTK's actual savings (if RTK is installed)
  rtk gain

THE JAVASCRIPT MINIFICATION ANALOGY
If you've worked with web development, you already know this
pattern. JavaScript minification strips whitespace, shortens
variable names, and removes comments before sending code to the
browser. The code does the same thing. The payload is smaller.
The browser is faster.

Token minimization is the same idea applied to LLM input.
Strip whitespace, compact JSON, remove noise. The information
is the same. The token count is smaller. The API call is cheaper.

The difference: JS minification saves bandwidth (download time).
Token minimization saves money (per-token pricing). In both
cases, you're compressing BEFORE the expensive operation — the
browser parsing JS, or the model processing tokens.

RTK is to LLM tokens what Webpack/Terser is to JavaScript
bundles. Same principle, different domain.

WHY MINIMIZATION MATTERS FOR AGENTS
In an agent loop, tool results accumulate in the conversation
history. Every prior tool result is re-sent as input on the
next turn. So a 200-token verbose result on turn 3 becomes
200 tokens of input on turns 4, 5, 6, 7... all the way to
the end. Minimizing it to 114 tokens saves 86 tokens PER
SUBSEQUENT TURN, not just once.

Combined with conversation caching (Part 1), minimized results
mean a smaller prefix to cache and read. The savings compound.

PART 5: count_tokens — Know Before You Spend

BACKGROUND
The API has a count_tokens endpoint (/v1/messages/count_tokens)
that counts tokens for a message WITHOUT generating a response.
Same parameters as messages.create — model, messages, system,
tools — but returns only {input_tokens: N}. No generation. No
output cost. Just the count.

The SDK exposes it as client.messages.count_tokens().

This is a free pre-flight check. The count itself costs nothing.

INPUT vs. OUTPUT — THE TWO SIDES OF THE BILL

When you call messages.create, you pay for two things:

  ┌────────────────────────────────────────────────────────────┐
  │ INPUT tokens (you pay to SEND)                            │
  │   system prompt + tool definitions + all messages          │
  │   → this is what count_tokens reports                      │
  │   → you know this BEFORE you call                          │
  │   → grows every turn (all prior messages re-sent)          │
  ├────────────────────────────────────────────────────────────┤
  │ OUTPUT tokens (you pay to RECEIVE)                         │
  │   the model's response text + thinking tokens              │
  │   → you CANNOT know this in advance                        │
  │   → depends on what the model decides to say               │
  │   → output pricing is 5x input across all models           │
  └────────────────────────────────────────────────────────────┘

The pricing per million tokens:

  ┌────────────────┬───────────┬────────────┐
  │ Model          │ Input $/M │ Output $/M │
  ├────────────────┼───────────┼────────────┤
  │ Haiku 4.5      │ $1.00     │ $5.00      │
  │ Sonnet 4.6     │ $3.00     │ $15.00     │
  │ Opus 4.6       │ $15.00    │ $75.00     │
  └────────────────┴───────────┴────────────┘

So count_tokens gives you the FLOOR — the minimum you will pay
for a call. The actual cost is that plus whatever the model
generates. Output is unpredictable and 5x more expensive per
token.

For agent loops, input is often the dominant cost anyway. The
context grows every turn (all prior messages re-sent as input),
while output per turn stays roughly constant. A 30-turn agent
loop re-sends 29 turns of history as input on the final turn.
The output is just the latest response.

WHY THIS MATTERS
  - Know the input cost floor BEFORE you send
  - See how the same text costs differently across models
  - Understand where the tokens actually go (system prompt vs.
    tools vs. user message)
  - Budget planning: input is predictable, output is not — plan
    for the part you can measure

THE FULL TOKENOMICS STACK
  1. Understand the costs (thinking accumulation, uncached prompts)
  2. Reduce costs (prompt caching, tool caching, conversation caching)
  3. Cap costs (token budget circuit breaker)
  4. Minimize input (compress text before it becomes tokens)
  5. Measure first (count_tokens before you send)

  These five layers turn "a credit card attached to a while loop"
  into a predictable, bounded, efficient line item.


WHY THIS MATTERS
  1. DENIAL OF WALLET: Without cost controls, an attacker who can
     trigger long agent loops (via injection, tool failures, or
     ambiguous tasks) burns your budget. Caching and thinking
     management are financial defense controls.

  2. PREDICTABLE COSTS: Understanding the token math lets you
     forecast agent costs accurately. System prompt size × turns ×
     model rate = baseline cost. Add thinking accumulation for
     reasoning-heavy workflows.

  3. SCALE MULTIPLIER: One agent is cheap. A hundred concurrent
     agents running 30 turns each on Opus without caching is not.

A NOTE ON BOUNCERS AND TOKENOMICS
The bouncer pattern (Demos 10 and 16) is first and foremost a
semantic gate — it validates inputs before they reach the tool.
But it has a tokenomics side effect worth noting.

When Pydantic rejects an invalid tool call BEFORE execution,
that's a call that never produces an error message, never
triggers a retry, and never adds noise to the context window.
The bouncer catches "server_id SRV-9999 not found" in code,
returns a clean validation error, and the model moves on.

Without the bouncer (Demo 11, Context Pollution), the same
invalid call produces a raw stack trace (400+ tokens), the
model retries, gets another stack trace, retries again. Each
retry adds tokens to the context. The cost grows quadratically.

So all context window management is tokenomics to some extent.
The bouncer is a security control first, but the cost savings
are a real and measurable side effect. Security and efficiency
aren't separate concerns — they're the same architecture.

Q&A PREPARATION

Q: "Is prompt caching automatic?"
A: No. You must add cache_control breakpoints to your API call.
   The default behavior is full-price processing every turn.

Q: "What if my system prompt is short?"
A: If it's below the minimum threshold (~1024 tokens), caching
   won't activate. But short prompts are cheap anyway — the savings
   matter most for realistic production prompts (1000+ tokens).

Q: "Should I always strip thinking tokens?"
A: Not always. For simple tool-calling agents, stripping is safe
   and saves money. For complex multi-step reasoning where the model
   builds on its prior analysis, test first. The right answer depends
   on your use case.

Q: "Does caching work across different conversations?"
A: Cache is keyed by the exact prompt prefix. Same system prompt
   across different conversations will share the cache if calls
   happen within the 5-minute TTL.

Q: "What should I set the token budget to?"
A: Depends on your task complexity and model. Profile a few typical
   runs, take the 95th percentile token count, add 20% headroom.
   Start generous and tighten as you learn your agent's patterns.
   The goal is catching runaways, not constraining normal operation.

Q: "How much does this actually save in production?"
A: Depends on turns and model. A 30-turn investigation on Opus
   with a 2K system prompt: caching saves ~$0.84 per run. With
   6 concurrent agents, that's ~$5/batch. Over a month of daily
   runs, it adds up to hundreds of dollars.

COST: ~$0.005-0.01 per run (Haiku, 3-5 turns)
